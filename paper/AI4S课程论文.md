# 基于大语言模型的企业年报智能提取Agent设计与实现

## 摘要

本文介绍了一个基于阿里云通义千问大模型的企业年报智能提取Agent。该系统能够从企业年报文本中自动提取资产负债表、管理层变动和附注信息，并输出结构化的JSON数据。系统采用纯前端架构，通过浏览器直接调用Qwen模型API，利用pdf.js解析PDF文件，通过四步工作流完成数据提取。最终部署在GitHub Pages上，可直接在线使用。测试结果表明，该Agent对年报关键数据的提取效果较好，能够满足基本的结构化需求。

**关键词：** 大语言模型；智能体；企业年报；信息提取

---

## 一、引言

### 1.1 选题背景

选这个题目其实是因为在学习过程中发现，企业年报的数据量很大，一份年报少则几十页多则上百页，如果靠人工去翻找资产负债表、管理层变动这些信息，效率非常低。而且不同公司的年报格式还不一样，有的用表格，有的直接写文字，用传统正则表达式去匹配的话，换一家公司可能就匹配不上了。

正好这学期学了AI4S课程，了解到大语言模型在文本理解方面很强，就想着能不能用LLM来做这件事。题目要求里第8题"企业年报数据格式化提取"正好符合这个方向，而且不需要用到图像识别或者物理仿真这些复杂的技术栈，比较适合课程作业的规模。

### 1.2 要解决的问题

简单来说就是：给定一份企业年报（文本或PDF），自动提取出三类关键信息——资产负债表的各科目数据、管理层人员变动情况、附注中的重要事项，并以结构化的JSON格式输出。

---

## 二、智能体相关概念

### 2.1 什么是智能体

课程里讲到智能体的核心公式是：**LLM + 工具 + 知识库 + 工作流**。我理解下来就是：

- **LLM** 是"大脑"，负责理解文本和做出判断
- **工具** 是"手脚"，比如PDF解析、API调用这些
- **知识库** 是"经验"，告诉LLM资产负债表长什么样、有哪些科目
- **工作流** 是"步骤"，把一个大任务拆成几个小任务按顺序执行

跟传统软件最大的区别在于，传统软件要写一堆if-else规则来处理不同格式，而智能体靠LLM的语义理解能力，天然就能适应各种格式变化。

### 2.2 为什么选这个方向

对比了几个题目之后选了第8题，主要考虑：

1. 纯文本处理，不需要GPU或者专业软件（不像StarCCM仿真那种）
2. 数据可以从公开渠道获取（上市公司年报都是公开的）
3. 技术栈比较清晰：前端 + LLM API，部署也简单
4. 能完整体现智能体的四个要素（LLM用Qwen，工具用pdf.js，知识库嵌入提示词，工作流分四步）

## 三、系统设计与实现

### 3.1 整体架构

系统是纯前端的，没有后端服务器。所有处理都在浏览器里完成，通过API调用阿里云的Qwen模型。选这个架构主要是因为：

- 部署简单，GitHub Pages免费托管静态文件
- 不用操心服务器运维
- 用户数据只在本地浏览器处理，不用上传到第三方

架构大致是这样的：

```
浏览器端
├── PDF解析（pdf.js）
├── 工作流引擎（四步提取）
├── LLM客户端（调用DashScope API）
└── 界面渲染（对话式UI）
        │
        ▼
阿里云DashScope → Qwen大模型
```

### 3.2 技术选型

| 模块 | 技术 | 选择理由 |
|------|------|----------|
| PDF解析 | pdf.js | Mozilla开源的，浏览器端直接用，不需要后端 |
| 大模型 | Qwen-plus | 阿里云的，课程要求用阿里云模型 |
| API协议 | OpenAI兼容格式 | DashScope提供的标准接口，文档比较全 |
| 前端 | 原生HTML/CSS/JS | 不想引入React/Vue这些框架，作业规模用不上 |
| 部署 | GitHub Pages | 免费，推代码自动部署 |

### 3.3 工作流设计

这是整个系统的核心部分。一开始我想的是让LLM一次性把所有信息都提取出来，但测试发现效果不好，容易漏信息。后来改成四步走的策略：

**第一步：文档预处理**
先识别出这是哪家公司、哪一年的年报、什么行业。这步看起来简单，但其实很重要，因为后面的提取需要知道上下文。比如同样叫"应付账款"，在制造业和金融业的含义可能不太一样。

**第二步：资产负债表提取**
这是最难的一步。资产负债表有大量数值数据，而且不同公司用的单位不一样（有的用元，有的用万元，有的用亿元），需要在提示词里明确要求统一转换。提示词里还要给出完整的科目结构，告诉LLM资产负债表应该包含哪些科目。

**第三步：管理层变动提取**
从年报的公司治理部分找人员变动。这步相对简单，主要是识别"辞职""任命""换届"这些关键词，然后提取人名、职位、变动类型。

**第四步：附注信息提取**
附注部分格式最不规范，有的公司写得很详细，有的就几句话。提示词里要定义好哪些类别是重要的（会计政策变更、关联交易、或有事项等），并要求LLM给每条附注打一个重要性标签。

每一步的输出都是JSON格式，方便下一步处理和最终汇总。

### 3.4 提示词工程

提示词的设计花了不少时间，因为LLM的输出质量很大程度上取决于提示词写得好不好。几个关键点：

1. **角色设定**：每步都给LLM一个具体角色，比如"专业的会计数据分析助手"，这样LLM会更有针对性地生成内容

2. **格式约束**：明确要求输出JSON，并给出完整的字段定义和示例。如果不给示例，LLM有时候会自由发挥，输出的字段名不一致

3. **温度参数**：设置成0.1，几乎不让LLM"创造"内容，尽量保证每次输出一致

4. **容错处理**：LLM输出的JSON有时候会有格式错误（比如多一个逗号），代码里做了多级容错——先尝试直接解析，不行就提取```json代码块，再不行就找第一个{和最后一个}

### 3.5 界面设计

界面做成了对话式的，类似ChatGPT的交互方式。主要是考虑到：

- 其他同学用Coze、Dify这些平台做的智能体都是对话式的，我做成一样的风格更符合课程要求
- 对话式的界面上下文连贯，用户能看到Agent的思考过程
- 工作流的执行状态直接嵌在对话里，不用切换页面去看

用户可以上传PDF或者直接粘贴文本，也可以先用内置的示例数据试一下。API Key的配置藏在右上角齿轮图标里，主界面不显示。

### 3.6 关键代码实现

**PDF文本提取**

pdf.js的getTextContent()返回的是扁平的文本项数组，需要按y坐标还原成段落。实现思路是：y坐标相近的（差值小于3）认为是同一行，同一行内按x坐标排序：

```javascript
const tolerance = 3;
const sorted = items.sort((a, b) => {
    const yDiff = b.transform[5] - a.transform[5];
    if (Math.abs(yDiff) > tolerance) return yDiff;
    return a.transform[4] - b.transform[4];
});
```

**流式输出**

DashScope支持SSE流式输出，用ReadableStream逐token读取。这样用户能实时看到LLM的生成过程，不用等全部生成完：

```javascript
const reader = response.body.getReader();
while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    // 解析SSE data行，提取delta.content
}
```

## 四、测试与评估

### 4.1 测试数据

准备了两份模拟的年报摘要，一份模拟万科（房地产行业），一份模拟腾讯（互联网行业）。之所以用模拟数据而不是真实年报，是因为真实年报太长，作为Demo展示不太方便。模拟数据覆盖了资产负债表的主要科目、管理层变动、附注信息等典型内容。

### 4.2 测试结果

| 提取任务 | 效果 | 遇到的问题 |
|----------|------|-----------|
| 公司基本信息 | 较好 | 行业分类偶尔不准 |
| 资产负债表 | 较好 | 个别科目名称匹配有偏差 |
| 管理层变动 | 一般 | 偶尔漏掉一两条变动 |
| 附注信息 | 一般 | 信息遗漏较多 |

总体来说，资产负债表的提取效果最好，因为结构比较规整。附注部分效果一般，主要是因为附注格式太自由了，有的公司写得很简洁，LLM容易漏掉。

### 4.3 遇到的问题和解决

1. **LLM输出的数值有误差**：有时候会四舍五入。解决办法是在提示词里强调"使用原文精确数值，不要修改"。

2. **不同公司科目名称不统一**：比如"货币资金"和"现金及银行存款"其实是同一个东西。目前没有特别好的解决办法，只能在提示词里尽量列举常见变体。

3. **长文本截断问题**：Qwen-plus有token限制，太长的年报需要分块处理。目前的方案是按段落分块，每块不超过6000字符。

## 五、讨论与心得

### 5.1 怎么做到"多快好省"

做完这个项目，有几点比较深的体会：

**任务拆分比什么都重要。** 一开始想让LLM一步到位，结果效果很差。后来拆成四步，每步只负责一个子任务，准确率明显提升。这其实跟写代码的"单一职责原则"是一个道理。

**提示词要反复调。** 提示词不是写一次就完事的，需要不断测试、修改。比如资产负债表的提示词，我调了大概五六版，才把输出格式稳定下来。关键是要给LLM足够的约束，但又不能太死板。

**前端优先，快速验证。** 纯前端架构的好处是改完代码刷新浏览器就能看到效果，不用等编译部署。GitHub Pages也很方便，push代码就自动更新。

**流式输出是必须的。** LLM生成需要时间，如果让用户干等着，体验很差。加上流式输出之后，用户能看到实时的生成过程，心理上感觉快很多。

### 5.2 局限性

这个系统还是有不少局限的：

- 数值精度不如OCR+正则的方案，LLM有时候会对数字"做手脚"
- 处理真实年报（几十页那种）效果会打折扣，因为需要分块，上下文会丢失
- API调用有成本，虽然新用户有免费额度，但大量调用还是会产生费用
- 不同公司的年报格式差异很大，提示词很难覆盖所有情况

### 5.3 可以改进的地方

如果以后继续做，可能会在这些方面改进：

1. 用RAG（检索增强生成）来处理长文本，先检索相关段落再让LLM提取
2. 加上结果校验，比如检查资产是否等于负债加权益
3. 支持批量处理，一次性分析多家公司的年报做对比
4. 用Qwen-VL处理年报中的图表部分

---

## 六、结论

本文实现了一个基于Qwen大模型的企业年报智能提取Agent，通过四步工作流（文档预处理→资产负债表提取→管理层变动提取→附注信息提取）实现了年报数据的自动化结构提取。系统采用纯前端架构，部署在GitHub Pages上，可直接在线使用。

通过这个项目，我对LLM智能体的开发有了比较实际的理解：任务拆分、提示词设计、容错处理是三个最关键的环节。LLM确实很强，但不是万能的，需要通过合理的工作流设计来弥补它的不足。
